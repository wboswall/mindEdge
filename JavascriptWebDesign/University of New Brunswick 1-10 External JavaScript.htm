Help, Hints, and How Did We Do That?

If you look at the index.html file, you will see we used <body onLoad="checkScript()" >. In this case, we wanted to check the capabilities of the browser, and onLoad allows us to do it somewhat seamlessly.

If we then search for our checkScript function in the script.js file, you will see a line that begins:

document.getElementById("p").innerHTML

This JavaScript code refers to the content between the <p> </p> tags, what is also known as innerHTML. Use innerHTML anytime you want to change what appears between two tags. Use outerHTML any time you want to change the tags as well.

For example, replace the second line of script.js with document.getElementById("p").outerHTML = "<h1>Congratulations you have correctly referencenced script.js</h1>"; and see what happens.

Using External JavaScript

The advantages of this approach are:

  • As is the case with external style sheets, this allows you to create one file that applies to many web pages. To make a change to all those pages, you only have to edit one file.

  • This approach keeps your HTML cleaner, which in turn, can make the job of content editors (who may be unfamiliar with JavaScript) much easier.

  • As a single file, your JavaScript can be cached by a browser, and this in turn, can make the overall use of the site appear to be faster.

Syntax and Names

JavaScript naming conventions tend to use what is called "camel case" a mix of upper- and lower-case. For example, a typical variable name might look like firstName. One of the challenges with JavaScript, as opposed to other languages, is that hyphens are not permitted (The "-" is reserved only for the minus operator). Underscores are acceptable, but the tendency is to favor either lower camel case (e.g., firstName) or upper camel case (e.g., FirstName), which is also called Pascal case since it is shared with the Pascal programming language.

When Not to Use External JavaScripts

In most cases, it's probably best to do an external JavaScript file rather than writing something inline. However, practicality sometimes dictates shortcuts.

Many web sites and web applications are tangled messes. You often hear the phrase spaghetti code* Spaghetti code is difficult to follow code, often reflective of having been written in a disjointed way by different developers.  to describe the conglomerations of HTML, style sheets, scripts, images, and other resources that produce someone's web presence. Whether you work as part of a team or a contractor, likely a lot of your web development work will involve picking up where some other developer left off.

The problem tends to arise because developers layer multiple technologies and snippets of code over what has been done in the past. As such, if you are trying to add one piece of interactivity, it may be easier to write one all-inclusive block of code (HTML, JavaScript, and CSS) rather than to have multiple files that have to be placed in the correct location. This is more a Band-Aid than a best practice.

By and large, you should keep your HTML as clean as possible, and an absolute red flag in any programming or scripting is when you find yourself writing the same or similar functions in two or more places.
